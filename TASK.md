# Техническое задание (ТЗ) на разработку проекта «PlanBot»

## 1. Общие положения

### 1.1. Наименование проекта
PlanBot — Telegram-бот для интеллектуального планирования задач.

### 1.2. Цель создания
Разработка автоматизированной системы (планировщика), которая принимает от пользователя неструктурированные или полуструктурированные записи о задачах и на основе встроенных алгоритмов приоритизации составляет оптимальный план распределения этих задач во времени с учетом личных временных ограничений пользователя.

## 2. Функциональные требования (Что система должна делать)

Система должна быть разделена на модули (Use Cases):

### 2.1. Модуль управления пользователями и профилем
*   **Регистрация/Авторизация:** Автоматическая при первом обращении к боту (идентификация по `user_id` Telegram).
*   **Настройка расписания пользователя (Working Schedule):**
    *   Пользователь указывает *рабочие* и *выходные* дни.
    *   Пользователь указывает *рабочее время* (например, с 09:00 до 18:00) и *нерабочее время* (задачи на это время не назначаются).
    *   Учет часового пояса пользователя (обязательно, иначе планирование сломается).

### 2.2. Модуль ввода задач (Inbox)
*   Пользователь отправляет боту сообщение с задачей.
*   Бот должен уметь парсить сообщение и запрашивать уточнения, если чего-то не хватает.
*   **Параметры задачи:**
    *   *Название/Описание* (обязательно).
    *   *Дедлайн (Due Date)* — конкретная дата/время, к которой нужно сделать.
    *   *Срочность (Priority/Urgency)* — пользователь может указать метку (Высокая, Средняя, Низкая).
    *   *Примерная трудоемкость (Effort)* — в часах или минутах (пользователь вводит число).
    *   *Тип времени (Time Type)* — возможность указать, что задачу можно делать только в рабочее или в любое время (личное).

### 2.3. Модуль планирования (Алгоритмическое ядро) — **Сердце проекта**
*   **Входные данные:** Список невыполненных задач с тегами (срочность, дедлайн, effort) + Расписание пользователя.
*   **Алгоритм ранжирования:**
    *   Задачи с ближайшим дедлайном получают наивысший приоритет.
    *   Если дедлайна нет, учитывается указанная пользователем срочность.
    *   Учет "стоимости" задачи (техника "помидора" или "съесть лягушку") — возможно мелкие задачи группировать.
*   **Алгоритм составления плана (Расписание):**
    *   Система берет свободные слоты в календаре пользователя.
    *   Назначает задачи в эти слоты, начиная с самых приоритетных.
    *   Учитывает, что задача может быть разбита на несколько дней, если она большая (например, задача на 8 часов не запихивается в один день, если пользователь работает 4 часа).
    *   На выходе получается "План" на неделю/месяц: список задач, привязанных к конкретному времени.

### 2.4. Модуль взаимодействия с пользователем (Telegram)
*   **Команды:**
    *   `/start` — приветствие, настройка профиля.
    *   `/add` или просто текст — добавить задачу (интерактивный ввод).
    *   `/plan` — сгенерировать план на день/неделю.
    *   `/today` — показать задачи на сегодня.
    *   `/done <ID задачи>` — отметить выполнение.
    *   `/settings` — изменить рабочее время.
*   **Интерактивность:** Использование Inline-клавиатур Telegram для подтверждения действий (например: "Я добавил задачу 'Написать отчет'. Верно ли указан дедлайн: 20.03? [Да] [Нет, изменить]").

### 2.5. Модуль интеграции с Google Calendar
*   **Авторизация:** Реализация OAuth 2.0 для доступа к Google Calendar пользователя.
*   **Синхронизация (Two-way):**
    *   **Push (PlanBot -> Google):** Утвержденный пользователем план из PlanBot должен создавать события (с пометкой "PlanBot" или "Задача") в Google Calendar.
    *   **Pull (Google -> PlanBot):** Импорт существующих событий из Google Calendar, чтобы PlanBot учитывал их как "занятое время" (не планировал поверх них новые задачи).

## 3. Нефункциональные требования (Как система должна работать)

### 3.1. Технологический стек (Stack)
*   **Язык:** Go (Golang) версии 1.21+.
*   **Telegram Bot API:** Библиотека `go-telegram-bot-api/telegram-bot-api/v5`.
*   **База данных:** PostgreSQL 15+ (основное хранилище).
*   **Миграции:** `golang-migrate/migrate`.
*   **Логирование:** `logrus` или `zap`.
*   **Конфигурация:** Через переменные окружения (12-factor app) + библиотека `viper`.
*   **Google API:** Официальный клиент `google.golang.org/api`.
*   **Кэширование (опционально):** Redis (для хранения сессий пользователей и временных данных, чтобы не долбить БД).

### 3.2. Архитектура
*   **Чистая архитектура (Clean Architecture)** или **Луковая архитектура** (Onion).
    *   **Слой 1: Domain (Модели):** Сущности User, Task, Schedule (не зависят ни от чего).
    *   **Слой 2: Use Cases (Сервисы):** TaskPlanner, UserManager (содержат бизнес-логику).
    *   **Слой 3: Infrastructure (Адаптеры):** Репозитории Postgres, Клиент Telegram, Клиент Google API.
*   Такой подход позволит легко заменить Telegram на другой мессенджер или добавить веб-интерфейс в будущем.

### 3.3. Требования к базе данных
*   **Миграции:** Все изменения схемы БД должны быть версионированы.
*   **Индексы:** Обязательно создать индексы по полям `user_id`, `status`, `deadline` для быстрой выборки плана.

### 3.4. Безопасность
*   Токены доступа к Google Calendar должны храниться в зашифрованном виде (например, с использованием AES).
*   Никаких секретов в коде — только переменные окружения.

## 4. Модели данных (Domain)

### 4.1. User (Пользователь)
```go
type User struct {
    ID           int64      // Telegram User ID
    TimeZone     string     // "Europe/Moscow"
    WorkDays     []time.Weekday // [1,2,3,4,5] (Пн-Пт)
    WorkStart    string     // "09:00"
    WorkEnd      string     // "18:00"
    GoogleToken  []byte     // Зашифрованный токен
    CalendarSync bool       // Включена ли синхронизация
    CreatedAt    time.Time
}
```

### 4.2. Task (Задача)
```go
type Task struct {
    ID          string    // UUID
    UserID      int64
    Title       string
    Description string    // Опционально
    Priority    string    // "high", "medium", "low"
    Effort      int       // Часы или минуты (нужно определиться с единицей)
    Deadline    *time.Time // Может быть nil
    Status      string    // "active", "completed", "deferred"
    CreatedAt   time.Time
}
```

### 4.3. Plan (План - результат работы алгоритма)
*Это может быть не таблица в БД, а просто объект в памяти или кэше, либо таблица `schedule_items`.*
```go
type PlanItem struct {
    TaskID     string
    UserID     int64
    StartTime  time.Time // Когда начать делать
    EndTime    time.Time // Когда закончить
    IsFromGoogle bool    // Заблокировано ли время событием из Google
}
```

## 5. Логика работы алгоритма (MVP)

1.  **Сбор:** Каждую ночь (или по команде `/plan`) система собирает все активные задачи пользователя.
2.  **Сортировка (Ранжирование):**
    *   Задачи сортируются по весу (Score). Формула может быть простой: `Score = (1 / (часы_до_дедлайна + 1)) * КоэфПриоритета`.
    *   Чем ближе дедлайн, тем выше задача.
3.  **Планирование (Жадный алгоритм):**
    *   Получаем список свободных окон из расписания пользователя на ближайшие N дней.
    *   Берем самую важную задачу.
    *   Ищем ближайшее окно, достаточное для размещения задачи (или ее части, если задача слишком большая).
    *   "Бронируем" это время под задачу.
    *   Повторяем, пока задачи не кончатся или не кончится свободное время.

## 6. Требования к разработке и поставке (DevOps)

*   **Docker:** Обязательно наличие `Dockerfile` и `docker-compose.yml` для локального запуска (бот + postgres + redis).
*   **CI/CD:** (Опционально, но желательно) Файл `.github/workflows` для запуска линтеров и тестов.
*   **Линтеры:** Использование `golangci-lint` в проекте.
*   **Makefile:** Наличие команд `make build`, `make run`, `make migrate`.

## 7. Этапы реализации (Roadmap)

1.  **Phase 0: Настройка проекта.** Структура папок, подключение к БД, миграции, базовый каркас бота (команда `/start`).
2.  **Phase 1: Ядро.** Модели User и Task, CRUD для задач в БД, настройка профиля пользователя (рабочее время).
3.  **Phase 2: Алгоритм.** Реализация простого планировщика (сортировка по дедлайну, назначение в свободные окна).
4.  **Phase 3: Telegram UI.** Команды `/add`, `/plan`, `/done`. Интерактивные диалоги.
5.  **Phase 4: Google Calendar.** Интеграция OAuth, импорт занятости, экспорт плана.
